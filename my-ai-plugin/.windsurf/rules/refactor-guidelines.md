# 代码拆分与模块归类规则

## 总原则
- 代码拆分优先按业务领域归类，而不是只按当前阶段、临时任务或一次性动作归类。
- 拆分的目标首先是让人更容易理解、定位和维护，而不是单纯把主文件行数拆小。

## 拆分决策顺序

### 1. 先按业务领域归类
- 同一领域的逻辑优先放在同一模块内，例如：`session`、`model`、`mode`、`runtimeState`、`webviewRouting`、`fileChanges`、`workspaceContext`。
- 后续继续拆分时，先检查现有同领域模块能否承接新逻辑，避免同一领域被分散到多个零碎 helper 文件中。
- 如果已有多个同领域 helper，优先考虑归并整理，再继续新增文件。

### 2. 再按职责边界归类
- 同一领域内，再按职责拆分，例如：
  - 会话数据加载/保存
  - 会话展示层访问
  - 会话列表响应构建
- 一个模块应尽量只承担一类稳定职责，避免同时混合“状态管理 + UI 响应 + 协议转换 + 外部 IO”。

### 3. 再按状态生命周期归类
- 不同生命周期的状态不要混放：
  - 单次请求级
  - 单轮生成级
  - 会话级
  - 全局应用级
- 会话级状态优先收拢在会话相关模块中，避免散落在多个阶段性 helper 中。

### 4. 再按依赖边界归类
- 纯逻辑、纯数据转换、纯响应构建优先独立为无副作用 helper。
- VS Code API、Webview 通信、文件系统访问等边界逻辑应单独归类。
- 主编排文件保留流程组织职责，不承载过多纯计算或结构转换细节。

### 5. 再按协议与领域的关系归类
- 响应构建、消息体整形、payload 转换可以独立抽离。
- 但如果这些响应构建明显属于某个领域，就优先并入该领域模块，而不是额外创建新的杂项文件。

### 6. 再按变化频率与稳定度归类
- 优先拆稳定、低耦合、低风险逻辑：
  - 纯函数
  - 列表构建
  - 状态读取/写入包装
  - 导出拼装
  - 响应构建
- 核心流式链路、复杂编排、跨轮状态回滚等高耦合逻辑，放在后面拆。

## 文件归并规则
- 同一领域的新逻辑优先并入已有同领域模块，不轻易再开同类新文件。
- 如果某个模块开始混入多个子领域，再拆成更明确的子模块。
- 不能因为阶段推进方便，就长期保留“临时落点式”文件结构。

## 命名规则
- 文件命名优先体现领域与职责，而不是阶段编号。
- 优先使用“原文件前缀 + 领域/职责”的命名方式，例如：
  - `ChatViewProvider_sessions`
  - `ChatViewProvider_runtimeState`
  - `ChatViewProvider_webviewRouting`
- 若同一领域必须继续细分，后缀应体现更具体职责，而不是单纯继续追加数字。

## 主文件保留规则
- 主文件保留：
  - 主流程编排
  - 关键状态流转
  - 高耦合核心链路
- helper 文件优先承接：
  - 纯计算
  - 数据整理
  - 响应构建
  - 低耦合状态访问
  - 边缘辅助能力

## 体量控制规则
- 单文件接近 1200 行时，必须继续评估拆分。
- 新 helper 也不能无限增长，避免把问题从一个大文件转移成多个“第二主文件”。
- 拆分后如果同领域 helper 变多，应定期回顾并归并。

## 执行规则
- 每轮优先做小步、低风险、可编译验证的重构。
- 每次拆分后都要检查：
  - 是否仍然按领域归类更清晰
  - 是否引入了新的职责混杂
  - 是否让调用关系更容易理解
- 如果只是把代码挪走，但没有让结构更清晰，这类拆分不算完成。
